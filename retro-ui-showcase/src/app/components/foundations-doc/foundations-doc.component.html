Colors

The retro-ui palette is derived from 14 semantic tokens. Never use raw hex in components.

Foundation Token Visual System

Overview

Colors in retro-ui are not picked individually. They flow from the Theme Builder contract: 14 primitive tokens per theme and mode, mapped to semantic aliases that components consume.

This guarantees that switching from classic-amber light to phosphor-green dark never breaks a component, because every color is resolved through a token.

Light vs Dark

The same semantic tokens resolve to different raw values depending on data-mode. A component using var(--panel) automatically adapts without extra CSS.

Light var(--panel) → warm paper
Dark var(--panel) → charcoal

Usage Guidelines

  • Always use semantic tokens (--panel, --text) in component styles.
  • Never hardcode hex colors; if a new color is needed, add it to the Theme Builder contract.
  • Accent color should be the only "loud" color; surfaces must stay neutral.
  • Feedback colors (--success, --danger, --info) are reserved for states and messages.

Accessibility

  • Minimum contrast ratio: 4.5:1 for normal text, 3:1 for large text.
  • Never communicate state by color alone. Pair --danger with an icon and a text message.
  • Ensure focus rings (--focus-ring) are visible against both --panel and --sunken.

Relation with Components

Every component in retro-ui consumes these tokens. retro-button uses --accent for hover, retro-input uses --input-bg and --line for borders, and retro-message maps severities directly to --success, --info, --danger.

Legacy Aliases

Typography

Two typefaces, one voice. Display for impact, monospace for structure.

Foundation Visual System

Overview

retro-ui uses a dual-font system. VT323 / IBM Plex Mono powers headings and chrome, while IBM Plex Mono handles body text, code, and data. This creates a terminal-like rhythm without sacrificing legibility.

Type Scale

Usage Guidelines

  • Avoid text smaller than 10px except for decorative badges.
  • Use --font-display sparingly: page titles, section headings, chrome.
  • Use --font-mono for body, labels, code, paths, and tabular data.
  • Keep --muted text above 4.5:1 contrast against its background.
  • Uppercase is reserved for labels, badges, and short headings. Avoid in long paragraphs.

Accessibility

  • Line height for body text should be at least 1.5 for readability.
  • Letter-spacing on small uppercase labels improves recognition.
  • Do not use font-weight alone to convey state; pair with color or icon.

Theme & Mode

Theme is identity. Mode is luminosity. Accent is emphasis. Three independent dimensions.

Foundation Token Theme

Overview

retro-ui separates visual identity from brightness. A theme defines the palette (classic-amber, phosphor-green, synthwave…). A mode defines whether that palette is rendered light or dark. An accent defines the highlight color.

Available Themes

@for (t of themes; track t.name) {
{{ t.label }} {{ t.name }} {{ t.identity }}
}

Theme Builder Contract

Every theme must define 14 primitive tokens for both light and dark modes. The semantic mapping is generic and shared. 

/* Surfaces */
--_desktop      /* page background */
--_panel        /* panel / card background */
--_panel-alt    /* alternate panel background */
--_sunken       /* recessed surface background */

/* Structure */
--_line         /* borders and structural lines */
--_line-soft    /* subtle dividers */

/* Text */
--_text         /* primary text */
--_muted        /* secondary / disabled text */

/* Form */
--_input-bg     /* form field background */

/* Accent palette */
--_accent       /* emphasis / highlight / interactive focus */
--_accent-brite /* bright accent (hover, active) */
--_success      /* positive / success */
--_info         /* informational */
--_danger       /* error / danger */

HTML Attributes

ThemeService writes three attributes to the <html> element: 

<html data-theme="classic-amber" data-mode="dark" data-accent="green">
  • data-theme switches the palette identity.
  • data-mode switches between light and dark surfaces.
  • data-accent overrides the highlight color (amber, green, cyan).

Usage Guidelines

  • Changing mode does not change theme, and vice versa.
  • Components consume semantic tokens; they never know which theme is active.
  • New themes must pass contrast checks for both light and dark variants.

Accessibility

retro-ui is built to be usable by everyone. Accessibility is a requirement, not a step.

Foundation A11y

Overview

The retro aesthetic is never an excuse for poor accessibility. Every component must be keyboard-navigable, screen-reader friendly, and visually contrasted.

Contrast

  • Normal text: minimum 4.5:1 against background.
  • Large text (18px+ or 14px bold): minimum 3:1.
  • UI components and graphical objects: minimum 3:1.
  • Focus indicators: minimum 3:1 against adjacent colors.

Keyboard Navigation

  • Tab moves focus to the next interactive element.
  • Shift + Tab moves focus backwards.
  • Enter / Space activates buttons, toggles checkboxes, opens selects.
  • Escape closes modals, dropdowns, and cancels operations.
  • Arrow keys navigate within menus, tabs, segmented controls, and sliders.
  • Home / End jump to first/last item in lists and tabs.

Screen Readers

  • All form controls have associated labels.
  • Dynamic content updates use aria-live regions (toasts, notifications).
  • Icons that convey meaning have aria-label or are hidden with aria-hidden="true".
  • Modals trap focus and restore it on close.

Motion & Animation

  • Respect prefers-reduced-motion. Disable non-essential animations for users who request it.
  • Loading skeletons and spinners are acceptable; avoid auto-playing motion that cannot be paused.

Do Not Rely on Color Alone

  • Error states include text and an icon, not just --danger color.
  • Success states include text and an icon, not just --success color.
  • Selected tabs use border + background + text weight, not just color.

Motion

Purposeful, restrained animation. Every transition must earn its place.

Foundation Motion

Overview

Motion in retro-ui is minimal and functional. Animations guide attention, confirm actions, and smooth state changes — but never distract or delay.

Transition Tokens

{{ m.name }} {{ m.value }} {{ m.desc }}

Motion Principles

  • Fast feedback: Hover and active states respond in under 100ms.
  • Predictable paths: Elements expand downward, modals fade in from center.
  • No decorative loops: Avoid infinite animations unless they indicate live status (pulse).
  • Respect reduced motion: Wrap transitions in @media (prefers-reduced-motion: no-preference).

Common Patterns

Hover background-color transition --transition-fast
Focus outline appearance --transition-fast
Expand / Collapse max-height + opacity --transition-slow
Modal / Toast opacity + translateY --transition-slow

Example: Building a Card

A project card in an application is not a retro-ui component. It is composed from retro-ui primitives: 

<retro-window title="Project" variant="default">
  <div style="display:flex;gap:var(--space-4);align-items:center">
    <retro-status-dot status="active" />
    <span style="font-family:var(--font-mono)">api-server</span>
  </div>
  <retro-tag label="v2.4.0" size="sm" />
  <retro-button variant="primary" size="sm">open</retro-button>
</retro-window>

The application owns the card. retro-ui owns the blocks.

Content Projection

Prefer slots over configuration props. A window with <div window-footer> is more flexible than a footerText input.

  • Use named slots for distinct regions (header, footer, actions).
  • Use default ng-content for the main body.
  • Avoid deeply nested slot trees; keep composition flat.

Usage Guidelines

  • If a pattern repeats in one app, create a local component; do not add it to retro-ui.
  • Keep retro-ui components domain-free. No Project, Task, or Dashboard concepts.
  • Test composed UIs in both light and dark modes.
